# 安全说明

## 当前安全设计

- 钱包签名只保存在当前浏览器会话的 `sessionStorage`。
- 钱包签名消息里绑定了 `appId`、`origin`、`sessionId`、`issuedAt` 和会话公钥。
- 对端在握手阶段会校验结构化签名，并继续做房间级 challenge-response。
- 每条消息使用会话临时私钥签名，由接收方使用握手验证过的会话公钥校验。
- 邀请导入后，会尽量从地址栏移除房间口令。
- 邀请带有效期，过期后拒绝继续加入。
- 消息文本通过 Vue 模板默认转义，不渲染 HTML。
- 消息长度和本地消息历史数量都有上限。
- 私聊房间限制最多一个远端成员。
- 房间口令、邀请链接和消息历史只保留在当前浏览器会话。
- `localStorage` 只保留最近房间索引，不含 `secret` 和消息正文。

## 风险边界

### 1. 这不是“匿名聊天室”

钱包地址本身就是身份标识。对端会知道你的地址。

### 2. 邀请链接是敏感信息

邀请中的 `secret` 就是房间口令。拿到链接的人，理论上都可以尝试加入房间。

建议：

- 通过安全渠道分发邀请。
- 不要把邀请贴到公开群或工单系统。
- 生产环境可以考虑邀请有效期和二次确认机制。

### 3. 本地设备仍然是安全边界

当前实现已经把房间口令、邀请链接、历史消息限制在 `sessionStorage`，但浏览器本地仍然会保存最近房间索引元数据，且前端内存中会存在活动会话材料。

建议：

- 仅在受信设备上使用。
- 不要在共享设备上长期保持登录标签页。
- 如果要上生产，仍然建议补本地加密、设备锁和更严格的会话清理策略。

### 4. 没有后端，意味着没有服务端安全兜底

没有以下能力：

- 服务端封禁
- 服务端审计
- 服务端历史恢复
- 服务端风控
- 服务端发放短期 TURN 凭据
- 把 TURN 使用权限真正限制到“只有你的网站”

进一步说：

- `STUN` 本身不是基于网站 origin 做访问控制的协议，不能安全地做到“只有我的前端页面能用”。
- 如果把 `TURN` 的固定用户名和密码直接写进前端环境变量，任何拿到页面代码的人都可以提取并复用它。
- 某些 origin 相关检查只能作为补充信号，不能当作真正的安全边界。

### 5. 会话密钥签名强于 HMAC，但仍不是链上消息签名

当前消息签名可以阻止房间成员直接伪造别人的消息，也能防止简单重放旧的登录 proof。但它依然不是每条消息都由钱包直接签名，因此不提供链上级别的不可抵赖性。

### 6. 只有 STUN 时无法保证所有网络都能连通

如果用户在对称 NAT、企业代理、严格防火墙环境中，只有 STUN 往往不够。

这不是代码漏洞，而是 WebRTC 网络现实。

### 7. TURN 可以提升连通率，但不等于“绝对可连”

当前实现支持 TURN 优先，再回退到 STUN / 混合模式。但最终连通效果仍取决于：

- TURN 服务是否可达
- TURN 账号配置是否正确
- 浏览器 ICE 策略
- 终端网络环境

### 8. relay 不是业务后端，但仍然是外部基础设施

当前实现通过外部公开 relay 做初始发现和握手。虽然消息主体走 P2P，但你仍然依赖公共网络基础设施。

## 生产建议

1. 增加消息签名或房间级别二次认证。
2. 为本地索引和会话历史增加可选加密存储。
3. 在生产环境中使用独立 TURN 账号和更安全的凭据轮换。
4. 为移动端增加更明确的权限提示和网络状态提示。
5. 评估是否提供自托管 relay。
6. 如果要把 TURN 使用权限真正收紧，增加一个最小凭据服务，为 coturn 发放短期凭据。
